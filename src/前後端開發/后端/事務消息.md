# 底層文件寸尺設計
●CommitLog:消息持久化的物理文件,所有隊列消息順序寫入同一個CommitLog文件,每 個Broker上的CommitLog被當前服務器上的所有ConsumeQueue共享。
●ConsumeQueue:消息順序寫入CommitLog 文件後將被異步轉發到邏輯隊列(ConsumeQueue)中,消費者可以消費邏輯隊列中的消息。ConsumeQueue中的消息結構包
含3個部分:該消息在CommitLog中的物理位置偏移量offset、消息實體內容的大小和 Message Tag 的hash值。
●IndexFile:Index 索引文件提供了對CommitLog 進行數據檢索的能力,可以通過key或者
時間區間來查找CommitLog中的消息。IndexFile在面試中基本不會出現,瞭解即可。


# 如何保证消息不丢失

- Producer端來說,如果消息未能持久化在Broker端,那麼消息是丟失的,消息的發送需要做好失敗重試(消費端需要冪等)。
- 從Broker端來說,保證消息不丟失需要選擇合適的刷盤策略(同步刷盤、異步刷盤)、主從複製策略(同步雙寫或者異步複製)。
- 從Consumer端來說,要確保消費者拉取到的消息被成功消費。
> - 消費失敗，並返回失敗狀態，那麼消息會再次被拉取並消費
> - 消息失敗，但返回失敗狀態時Broker宕機,那麼Consumer會定時重試。
> - 如果Consumer和Broker一起宕機,那麼由於Consumer消費的offset是定時異步持久化的, 因此Consumer和Broker重啓後,消息不會丟失但可能產生重複消費。

從全局來看,保證消息不丟失可能存在重複消費的情況,需要業務方冪等重試。

●解決消息不丟失問題,你覺得下面兩種方案哪種更靠譜?方案一:完全依賴消息隊列
保證消息不丟失。方案二:生產者發送消息失敗後重試,確保成功與消費者冪等消費,
既能保證消息隊列的高吞吐又能保證業務消息的不丟失。
